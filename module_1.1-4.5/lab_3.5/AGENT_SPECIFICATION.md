# AGENT_SPECIFICATION.md (lab_3.5)

## 1. Goal (цель агента)
Каждые 5 минут проверять доступность API Open-Meteo и логировать результат проверки.

Проверяемый эндпоинт (пример):
https://api.open-meteo.com/v1/forecast?latitude=55.7558&longitude=37.6173&current_weather=true

---

## 2. Метрики успеха (Success Metrics)

API считается доступным, если:
- HTTP статус ответа = 200 OK
- ответ содержит поле `current_weather`
- время ответа меньше 1000ms

---

## 3. Действия при событиях

### Успех (Success)
- записать в лог:
  - timestamp
  - statusCode
  - responseTime
  - message: "OK"

### Неудача (Fail, статус != 200)
- записать в лог:
  - timestamp
  - statusCode
  - message: "Bad status code"

### Критическая ошибка (Critical error)
Ситуации:
- timeout
- нет сети
- ошибка DNS
- ошибка парсинга JSON

Действие:
- записать в лог подробности ошибки
- вывести в консоль: `ALERT! API is down`

---

## 4. Состояние (Memory)

Агент должен помнить между запусками:
- lastSuccessTime (время последней успешной проверки)
- consecutiveFails (количество подряд неудачных проверок)
- lastAlertTime (чтобы не спамить алертами)

Пример логики:
- если подряд 3 ошибки → поднять ALERT
- если ALERT уже был меньше 10 минут назад → не повторять

---

# Архитектура агента (компоненты)

## 1. Scheduler (планировщик)
Отвечает за запуск проверки каждые 5 минут.
Инструмент: node-cron или setInterval.

---

## 2. Probe (зонд)
Функция, которая делает HTTP-запрос к API.

Требования:
- использовать timeout (например 3000ms)
- возвращать statusCode, responseTime, data
- ловить ошибки сети

Инструмент: axios (удобнее управлять timeout и ошибками).

---

## 3. Analyzer (анализатор)
Проверяет результат Probe:
- statusCode == 200
- наличие нужного поля current_weather
- responseTime < 1000ms

Возвращает итоговый статус:
- OK
- FAIL
- CRITICAL

---

## 4. Logger (логгер)
Записывает результаты проверки в файл monitor.log.

Формат: JSON Lines (одна строка = один JSON объект).
Пример строки:
{"time":"2026-04-18T12:00:00Z","status":"OK","statusCode":200,"responseTime":320}

Инструмент: pino (логгер пишет сразу JSON, удобно анализировать).

---

## 5. Action Dispatcher (диспетчер действий)
Принимает результат Analyzer и решает что делать:
- OK → просто лог
- FAIL → лог с деталями
- CRITICAL → лог + console ALERT

Также проверяет memory:
- если слишком много ошибок подряд → alert
- если alert уже был недавно → не спамить

---

# Выбор инструментов и почему

## HTTP клиент: axios
Причина:
- удобный timeout
- понятные ошибки
- проще чем fetch в Node.js

## Планировщик: node-cron
Причина:
- стабильный запуск по расписанию
- понятный формат cron (каждые 5 минут)

## Логгер: pino
Причина:
- быстрый и простой
- сразу JSON формат
- удобно читать и парсить в будущем
