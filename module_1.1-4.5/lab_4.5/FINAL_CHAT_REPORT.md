# Финальный отчёт: безопасный чат с LLM (тема 5)

## Блок 2 — промпт для генерации прокси (приложить к заданию)

```text
Создай файл server.js для Express-сервера (Node 18+, ES modules: "type": "module" в package.json), который выступает прокси для OpenAI API. Используй пакет openai v4+.

Требования:
1) Загружать OPENAI_API_KEY из dotenv (.env), не логировать ключ.
2) POST /api/chat, тело JSON: { message: string, history?: Array<{role:'user'|'assistant', content:string}> }.
3) Перед сообщениями пользователя добавляй system: "Ты — полезный ассистент в интерфейсе образовательной платформы 'CodeHub'. Отвечай кратко и по делу."
4) Вызов openai.chat.completions.create с model: gpt-3.5-turbo, messages, stream: true.
5) Ответ клиенту — SSE: Content-Type: text/event-stream, каждый чанк в виде строк data: {"content":"..."}\n\n, в конце data: {"done":true}\n\n.
6) Ошибки: до отправки заголовков — JSON 400/500; после начала SSE — data: {"error":"..."}\n\n и завершение.
7) Подключи cors и express.json с разумным лимитом тела.

Добавь package.json с зависимостями express, dotenv, cors, openai и скрипт "start": "node server.js".
```

**Вывод по стримингу:** ИИ обычно корректно предлагает `stream: true` и цикл `for await`, но часто путает формат отдачи клиенту (сырой текст вместо SSE, забытые `\n\n`, неверная обработка ошибок после `res.write`). Нужна ручная проверка через `curl -N` или Postman и сверка с парсером на фронте.

## Блок 3 — промпт для ChatWidget (React)

```text
Создай компонент ChatWidget.jsx для React 18:
UI: плавающая кнопка открытия/закрытия, окно чата, скроллируемый список сообщений, поле ввода, кнопка отправки, кнопка «Новый диалог» сбрасывает массив сообщений.
Состояние: messages: {role:'user'|'assistant', content:string}[], input, loading, open.
sendMessage(): POST на http://localhost:3000/api/chat с { message: текст, history: messages до добавления текущего пользовательского } — после отправки добавь в UI сообщение user и пустое assistant, читай response.body через getReader(), парси SSE (строки data: {...}), дописывай content в последнее сообщение assistant.
Стили: инлайн-стили или минимальный CSS, контрастные кнопки.
Доступность: aria-label на кнопках отправки и переключения чата, aria-live polite на ленте, role="dialog" на окне.
```

### Анализ UX (два принципа)

1. **Обратная связь (соблюдён):** во время запроса показывается состояние загрузки («Ассистент печатает…»), текст ответа наращивается по мере стрима — пользователь видит прогресс, а не «пустой экран».
2. **Доступность (частично соблюдена):** добавлены `aria-label`, `role="dialog"`, `aria-live` на логе; **риск нарушения** — частые обновления стрима в `aria-live` могут быть навязчивы для скринридеров; в production стоит ограничить анонсы (например, live только по завершению фразы или кнопка «прочитать ответ»).

## Блок 4 — нагрузочный мысленный эксперимент (1000 одновременных пользователей)

**Слабые места текущей учебной архитектуры:**

1. **Один инстанс Express без балансировки и очередей** — исчерпание CPU/RAM, рост задержек и обрывов SSE; лимиты ОС на открытые соединения.
2. **Квоты и биллинг OpenAI** — всплеск параллельных запросов упирается в rate limit провайдера; при ошибках клиенты получают деградацию без graceful backoff.

**Как развивать:** горизонтальное масштабирование за reverse-proxy, очередь задач (worker + Redis), кэширование идемпотентных ответов, circuit breaker, отдельный edge для статики, мониторинг p95 latency и 5xx.

## Архитектурный итог

Даже «простой» AI-чат почти всегда **fullstack**: секрет нельзя отдавать в браузер, нужны **валидация**, **политика доступа**, **учёт токенов** и **стабильный канал стрима** — это задачи доверенного сервера. Чистый фронт подходит разве что для публичных демо без секретов или для локального Ollama в доверенной среде, но не для типичного облачного API.

---

## Оценка сложности промптов: UI vs потоковый прокси

| Задача | Сложность промпта | Где ИИ силён | Где нужны правки человека |
|--------|-------------------|--------------|---------------------------|
| UI-виджет | Средняя | Разметка, состояние, базовый fetch | Стриминг + граничные случаи (обрыв, частичный JSON) |
| Прокси + SSE | Выше | Шаблон Express + OpenAI SDK | SSE формат, ошибки после заголовков, CORS, лимиты тела |

## Самая неочевидная техническая проблема

**Корректный парсинг SSE на фронте:** чанки `ReadableStream` не совпадают с границами событий — буферизация до `\n\n` обязательна. Второй частый затык — **CORS** (preflight на POST, заголовки стрима): решается явным `cors()` на прокси и отсутствием лишних кастомных заголовков на первом этапе.

## Роль разработчика в AI-First продукте

Это **сборная роль**: **архитектор системы** (границы клиент–сервер–модель), **интегратор** (контракты API, стриминг), **промпт-инженер** (system/developer политика), **специалист по безопасности и устойчивости** (секреты, rate limit, PII) и **инженер качества** (метрики, стоимость токенов, UX ошибок). Генерация кода ИИ ускоряет черновик, но ответственность за архитектуру и риски остаётся на человеке.
