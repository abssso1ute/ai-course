# Архитектура чата с LLM (учебный fullstack)

## Выбор провайдера и подхода

| Вариант | Суть | Когда уместен |
|--------|------|----------------|
| **A — облачный API** (OpenAI, Yandex GPT, GigaChat и т.д.) | Ключ на сервере, высокое качество, плата за токены | Боевой продукт, быстрый старт |
| **B — локально (Ollama)** | Данные не уходят наружу, нагрузка на железо | Строгая приватность, офлайн |
| **C — mock / демо** | Фиксированные ответы без LLM | Отладка UI, курс без ключей |

В реализации `chat-backend` заложен **вариант A** (OpenAI SDK); при отсутствии ключа для учёбы можно временно подменить ответ в прокси на mock, не меняя контракт `POST /api/chat`.

## Компоненты системы

### Фронтенд (React; аналогично строится на Vue 3)

- **`ChatWidget.jsx`** — плавающая кнопка открытия/закрытия, окно диалога, скроллируемая лента сообщений, поле ввода, отправка, индикатор загрузки/стриминга, кнопка **«Новый диалог»** (сброс истории на клиенте).
- **`MessageBubble.jsx`** (логически выделяемый блок внутри виджета или отдельный файл) — отрисовка одного сообщения (`role`, `content`), визуальное различие user/assistant.

Интеграция: корневой layout SPA (например, `App.vue` / `App.jsx`) рендерит `<ChatWidget />` один раз, чтобы виджет был доступен на всех маршрутах.

### Бэкенд (обязательный прокси на Node.js + Express)

Папка: **`lab_4.5/chat-backend`**.

Роли сервера:

1. **Секреты:** хранить `OPENAI_API_KEY` только в `.env` на сервере; клиент никогда не получает ключ.
2. **Валидация:** проверять тело запроса (`message`, опционально `history`), ограничивать размер/формат, отсекать мусор.
3. **Политика модели:** добавлять **system prompt** (роль ассистента платформы), собирать массив `messages` для Chat Completions.
4. **Streaming:** вызывать API с `stream: true` и пересылать клиенту **SSE** (`Content-Type: text/event-stream`), чтобы UI обновлялся по мере генерации.
5. **Ошибки:** при сбоях до начала стрима — JSON с кодом; после `text/event-stream` — событие с `{ error }` в том же протоколе, где возможно.

## Поток данных

```text
Пользователь вводит текст
    → Фронтенд (fetch POST /api/chat, body: { message, history })
        → Прокси Express
            → OpenAI Chat Completions (stream)
        ← чанки токенов
    ← SSE: data: {"content":"..."} / завершение
    → UI дописывает ответ ассистента
```

## Почему нельзя встраивать API-ключ в клиентский JavaScript

1. **Публичная утечка:** любой может открыть DevTools → Sources/Network, скачать бандл или перехватить строку с ключом; ключ считается скомпрометированным.
2. **Финансовый и репутационный ущерб:** злоумышленник вызывает API от вашего имени (массовые запросы, дорогие модели) — **неконтролируемые списания** и блокировка аккаунта у провайдера.
3. **Невозможность нормальной политики доступа:** на клиенте нельзя надёжно ограничить **rate limit**, **квоты**, **IP**, **аутентификацию пользователей**; ключ даёт полный доступ к вашему биллингу, тогда как сервер может проверять сессию, роли и обрезать запросы.
