Архитектура агента для проверки зависимостей
1. План выполнения (workflow агента)

Я бы разбил работу агента на такие шаги:

Получение доступа к репозиторию
Аутентификация через GitHub API (токен доступа)
Выбор нужного репозитория
Извлечение зависимостей
Считать package.json
Получить список dependencies и devDependencies
Проверка уязвимостей
Для каждой зависимости отправить запрос в npm audit API или Snyk API
Получить список известных уязвимостей и их severity
Формирование отчёта
Собрать результаты в структурированный формат (например: название пакета → версия → риск → описание проблемы)
Отправка в Slack
Отправить итоговый отчёт через Slack Webhook в нужный канал
2. Какие внешние API нужны агенту
GitHub API — доступ к репозиторию и файлам
npm audit API или Snyk API — проверка уязвимостей зависимостей
Slack Webhook API — отправка отчёта в канал
(опционально) GitHub Actions — для автоматического запуска по расписанию
3. Возможные ошибки и проблемы на этапах

1) GitHub API

ошибка авторизации (невалидный токен)
отсутствие доступа к приватному репозиторию
лимиты rate limit API

2) Чтение package.json

файл отсутствует или повреждён
нестандартная структура проекта

3) Проверка уязвимостей

зависимость не найдена в базе
API временно недоступен
разные версии одной библиотеки дают разные результаты

4) Формирование отчёта

пустой список зависимостей
слишком большой объём данных

5) Slack

неверный webhook URL
превышение лимита сообщения
неправильный формат payload
4. Как агент должен это учитывать
retry-механизм при ошибках API
fallback-логика (например, если Snyk недоступен → использовать npm audit)
валидация входных данных на каждом шаге
логирование всех этапов (чтобы можно было отладить)
защита от пустых или некорректных ответов API
