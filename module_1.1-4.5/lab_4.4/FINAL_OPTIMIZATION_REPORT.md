## Качественный анализ

**Самая сложная для решения с помощью ИИ задача** в нашем цикле оптимизации — **согласованное code splitting под реальную маршрутизацию и граф зависимостей**. ИИ уверенно предлагает шаблон `React.lazy` + `Suspense` или `defineAsyncComponent` во Vue, но часто не видит **скрытые статические импорты** (тяжёлая библиотека тянется через общий `index.ts`), **общие провайдеры** и порядок загрузки layout vs страницы. Из‑за этого сборка либо не уменьшает initial chunk, либо появляются «мигания» и ошибки границ Suspense. Потребовалась ручная проверка графа импортов и `stats.html`.

Вторая зона трения — **кэширование и заголовки**: рекомендации ИИ бывают общими; для конкретного хостинга (Vercel `vercel.json`, Netlify `_headers`) нужно сверять пути к **хешированным** файлам (`/assets/*`), поведение HTML (`no-cache`) и то, как платформа сама отдаёт статику.

## Роль ИИ в оптимизации (оценка 1–10)

| Роль | Оценка | Комментарий |
|------|--------|-------------|
| **Перфоманс-аудитор** (чеклисты, гипотезы, объяснение метрик) | **8 / 10** | Хорошо структурирует «что проверить» и типовые причины регрессий; риск — устаревшие советы или неверный LCP-кандидат без просмотра trace. |
| **Инженер по оптимизации** (патчи в код и конфиги) | **7 / 10** | Быстро выдаёт черновики (lazy, picture, preload, конфиги кэша); слабее в уникальной архитектуре репозитория, edge cases сборщика и валидации «действительно ли чанк отделился». |

**Сильные стороны:** шаблоны оптимизаций, объяснение trade-off, заготовки для Vite/Webpack, тексты для PR.

**Слабые стороны:** контекст репо, актуальные версии плагинов, полевые данные CrUX, специфика PaaS.

## Вопрос блока 3: какая продвинутая техника важнее для пользователей с медленным интернетом?

На **первом заходе** наибольший эффект даёт связка **разделение кода (route-based / component lazy)** и снижение **initial JS/CSS**: на узком канале время пропорционально объёму загружаемых байт и числу парсингов на main thread; меньший первый бандл быстрее приводит к интерактивности и снижает конкуренцию за сеть с LCP-ресурсами.

**Агрессивное кэширование** хешированной статики (`Cache-Control: public, max-age=31536000, immutable`) даёт огромный выигрыш на **повторных визитах** и при навигации по уже скачанным чанкам, но не убирает стоимость **первой** загрузки.

**Preload** полезен точечно для **известного LCP-ресурса** (hero, критический шрифт WOFF2), но при ошибочном выборе объекта может **обокрасть** полосу у более важного ресурса. Итог: для «медленный интернет, первый заход» в приоритете **уменьшение и разбиение загружаемого кода**; **кэш** — обязателен для устойчивого UX при возвратах; **preload** — хирургически под подтверждённый LCP.

## Три обязательных правила для ранних промптов в следующем AI-First приложении

1. **Явные требования к загрузке и LCP:** в промпте на каждый медиа-компонент — `width`/`height`, форматы WebP/AVIF + fallback, `loading` (eager только для LCP), при необходимости `fetchpriority` и согласованный `preload` в шаблоне `index.html`.

2. **Маршруты и чанки с первого дня:** генерировать страницы так, чтобы тяжёлые участки (графики, редакторы, AI-панели) жили за **dynamic import** / отдельными чанками; запретить «тихий» импорт тяжёлых библиотек в общий entry.

3. **Измеримость встроена в процесс:** с первого спринта — скрипт **анализа бандла** и порог на размер initial chunk; для UI с чатом — бюджет на **INP** (виртуализация списка, не блокировать main thread тяжёлым markdown без воркера).
