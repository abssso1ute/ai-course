# Анализ предметной области — городской транспорт и дашборд

## Назначение системы

Оперативный дашборд для диспетчеризации и аналитики наземного городского транспорта: отображение положения ТС, загруженности маршрутов и остановок, ключевых показателей эффективности. Данные поступают в режиме, близком к реальному времени, и из исторических хранилищ для отчётов и сравнений.

## Ключевые сущности

### Vehicle (Транспортное средство)

| Атрибут | Описание |
|--------|-----------|
| `id` | Уникальный идентификатор ТС в учётной системе |
| `fleetNumber` | Бортовой / гаражный номер (для отображения диспетчеру) |
| `type` | Тип: `bus`, `trolleybus`, `tram` (влияет на иконку и фильтры) |
| `routeId` | Связь с маршрутом, на котором сейчас работает ТС |
| `tripId` | Текущий рейс (связь с Trip) |
| `position` | Географические координаты и направление (`lat`, `lng`, `bearingDeg`) |
| `delayMin` | Отклонение от расписания в минутах (отрицательное — опережение) |
| `speedKmh` | Мгновенная или усреднённая скорость |
| `occupancy` | Заполненность (опционально, если есть данные АСКП) |
| `lastUpdatedAt` | Время последнего обновления позиции |

### Route (Маршрут)

| Атрибут | Описание |
|--------|-----------|
| `id` | Внутренний идентификатор |
| `number` | Номер маршрута для людей (например, `555`) |
| `name` | Название направления |
| `path` | Геометрия линии (полилиния в WGS-84), упрощённый аналог shapes в GTFS |
| `stops` | Упорядоченный список остановок маршрута |

### Stop (Остановка)

| Атрибут | Описание |
|--------|-----------|
| `id` | Идентификатор остановки |
| `name` | Название для UI и запросов на естественном языке |
| `lat`, `lng` | Координаты |
| `routes` | Маршруты, проходящие через остановку |

### Trip (Рейс)

| Атрибут | Описание |
|--------|-----------|
| `id` | Идентификатор рейса |
| `routeId` | Маршрут |
| `vehicleId` | Назначенное ТС |
| `scheduledStart`, `scheduledEnd` | Плановые границы рейса |
| `actualStart` | Фактический выход (для расчёта опозданий) |

Связи: **Route** объединяет **Stop** в последовательности; **Vehicle** выполняет **Trip** по **Route**; обновления позиций **Vehicle** приходят потоком.

## Источники данных (учебный проект)

1. **Мок API + WebSocket (`mockBackendServer.js`)**  
   Имитирует поток координат 50–100 ТС по заданным маршрутам, REST для снимков и исторических агрегатов. Форматы полей сознательно близки к идеям **GTFS** (маршруты, остановки, форма линии) и **GTFS-RT** (обновления `position`, `delay`), без полной совместимости со стандартом.

2. **История (встроенная в мок-сервер)**  
   Генерируются синтетические временные ряды загрузки по маршрутам и остановкам за выбранный период (для аналитика и графиков).

3. **Роли**  
   - **Диспетчер:** доступ к карте и метрикам в реальном времени.  
   - **Аналитик:** те же компоненты + запросы к истории и построение графиков через UI и чат-агента.

Реальная интеграция с ЕСМТ / поставщиками NAV могла бы заменить мок без изменения контрактов на уровне фронтенд-сторов.
