# Техническое задание — транспортный дашборд и чат-аналитик

## Цели и ограничения

- SPA с картой, графиками, панелью KPI и чатом.
- Режим реального времени через WebSocket (с fallback на опрос REST).
- Роли: диспетчер и аналитик (переключатель в UI влияет на доступные действия и подсказки).
- Чат формирует **структурированные команды** для стора дашборда, а не только текст.

## User Stories

### Диспетчер

1. **Как диспетчер**, я хочу видеть на карте движущиеся ТС разных типов своими значками, чтобы быстро отличать автобусы, троллейбусы и трамваи.
2. **Как диспетчер**, я хочу видеть задержку рейсов в минутах и список сильно опаздывающих ТС, чтобы перераспределять ресурсы.
3. **Как диспетчер**, я хочу фильтровать карту по типу ТС одним запросом в чате («оставь только троллейбусы»), чтобы сократить шум на экране.
4. **Как диспетчер**, я хочу видеть ключевые метрики (ТС на линии, средняя скорость, доля рейсов в графике), чтобы оценить ситуацию за секунды.

### Аналитик

5. **Как аналитик**, я хочу спросить у бота: «Построй график загрузки маршрута №555 за последнюю неделю», чтобы увидеть временной ряд на дашборде без ручной настройки фильтров.
6. **Как аналитик**, я хочу сравнить пассажиропоток на двух остановках столбчатой диаграммой по команде из чата, чтобы подготовить отчёт.
7. **Как аналитик**, я хочу спрятать все графики и оставить только карту командой «спрячь графики», чтобы сфокусироваться на геометрии движения.

### Общие

8. **Как пользователь**, я хочу ограничить карту географическим сектором («северо-запад») через чат, чтобы анализировать район.
9. **Как пользователь**, я хочу получать текстовый отчёт после действия («показано N трамваев»), чтобы подтвердить, что система поняла запрос.

## Функциональные требования (сводка)

| Область | Требование |
|---------|------------|
| Карта | Маркеры ТС, разные иконки по `type`, обновление позиций без полной перерисовки SPA |
| Графики | Временные ряды и гистограммы из данных стора / API |
| Чат | Классификация намерения, извлечение сущностей, команды в `dashboardStore` |
| API | REST: снимок ТС, история по маршруту/остановкам; WS: поток обновлений |

## Нефункциональные требования

- При 200+ маркерах карта должна оставаться отзывчивой (императивное обновление маркеров; опционально кластеризация при отдалении).
- Смена временного диапазона на графиках не должна блокировать UI (debounce / лёгкая перерисовка Chart.js).

## Критерии приёмки (учебная версия)

- Запуск `npm run dev` + `npm run server` открывает дашборд с живой картой и метриками.
- Чат понимает минимум: фильтр типа ТС, фильтр района, построение графика по маршруту, скрытие панелей графиков, запрос опозданий > N минут.
- В репозитории есть описанные артефакты DOMAIN / архитектура / отчёт.
