# Презентация (5–7 слайдов) — Smart Parking

Тезисы для Google Slides / PowerPoint / Keynote.

---

## Слайд 1 — Проблема и ценность

- Городские и логистические парковки теряют выручку из-за **непрозрачной занятости** и сложного UX.
- **Smart Parking** даёт водителю карту + быстрый сценарий брони, администратору — загрузку и тарифы.
- **AI-first** чат снижает порог входа: не нужно изучать фильтры.

---

## Слайд 2 — Решение (MVP)

- SPA: **Vue 3 + Pinia + Vite**, карта 8×12, статусы мест, одно активное бронирование.
- Ассистент на **русском**: поиск у входа, «дешевле», бронь, отмена, вопрос о брони.
- Мок **датчиков** (обновление каждые 10 с) имитирует реальный поток событий.

---

## Слайд 3 — AI-first архитектура

- **Не** «чат ради чата»: `agentCore` → структурированное намерение → **оркестратор** → Pinia → UI (подсветка + CTA).
- Внешняя LLM **не обязательна** в MVP: детерминированные правила + тесты.
- Зарезервирован контур **LLM-прокси** для сложного NLU и мультиязычности.

---

## Слайд 4 — Ключевые технические решения

- **Pinia** как единый источник истины; чат только вызывает разрешённые действия.
- **Оптимизация карты:** `v-memo` на ячейках при частых обновлениях.
- **Code splitting:** ленивый `/admin` (`import()`).
- **Service Worker** для кэша shell и офлайн-fallback.
- **Playwright** E2E на сквозной сценарий.

---

## Слайд 5 — Качество: тесты и метрики

- E2E: «открыть → найти через чат → забронировать → подтверждение».
- Lighthouse: цели — **LCP** (первый экран), **INP** (чат/карта), **CLS** (стабильность панелей).
- Скриншоты «до/после» — в `lighthouse/` (см. README там).

## Слайд 7 (опционально) — Риски и следующий шаг

- Риски: ошибочное намерение, приватность чата, зависимость от провайдера LLM.
- Дальше: реальные датчики/WebSocket, CV номеров, серверные политики и мультиагентность для сети парковок.
