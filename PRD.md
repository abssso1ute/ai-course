# Smart Parking — Product Requirements Document (PRD)

## 1. Контекст и цель

Веб-SPA для управления «умной» парковкой: карта мест, бронирования, админ-функции и **AI-first** чат-ассистент, который понимает запросы на естественном языке и вызывает действия в приложении (поиск, подсветка, бронь).

**Целевая аудитория:** водитель (поиск/бронь), администратор (статистика, тарифы, места).

## 2. Объём продукта (MVP)

- Интерактивная карта с статусами: свободно / занято / забронировано.
- Список свободных мест и детали активного бронирования водителя.
- Чат-ассистент: поиск места по критериям (вход, цена), бронь на время, отмена, вопрос «где моё бронирование».
- Админ: статистика загрузки, тарифы (базовая модель), добавление/удаление мест (мок).
- Мок-API датчиков: периодическое обновление статусов 1–2 мест.

## 3. Пользовательские сценарии (User Stories)

### Водитель

| ID | User Story |
|----|------------|
| D-01 | **Как** водитель, **я хочу** видеть карту парковки с цветовой разметкой статусов, **чтобы** быстро оценить занятость. |
| D-02 | **Как** водитель, **я хочу** видеть список свободных мест с ценой, **чтобы** выбрать подходящее без чата. |
| D-03 | **Как** водитель, **я хочу** сказать ассистенту: «Найди мне свободное место рядом с входом №3», **чтобы** система нашла и **выделила** кандидата на карте. |
| D-04 | **Как** водитель, **я хочу** сказать: «Забронируй место на 2 часа с 15:00», **чтобы** оформить бронь с подтверждением в чате и в блоке «Моё бронирование». |
| D-05 | **Как** водитель, **я хочу** сказать: «Паркуйся дешевле», **чтобы** ассистент предложил свободные места с **минимальным тарифом** в данный момент. |
| D-06 | **Как** водитель, **я хочу** нажать «Забронировать» после ответа ассистента, **чтобы** зафиксировать выбранное место без повторного ввода. |
| D-07 | **Как** водитель, **я хочу** сказать: «Отмени бронирование», **чтобы** освободить место, если планы изменились. |
| D-08 | **Как** водитель, **я хочу** сказать: «Где моё бронирование?», **чтобы** увидеть детали (место, время окончания). |

### Администратор

| ID | User Story |
|----|------------|
| A-01 | **Как** администратор, **я хочу** видеть статистику загрузки (доля занятых/свободных/забронированных), **чтобы** планировать ценообразование. |
| A-02 | **Как** администратор, **я хочу** задавать базовый тариф (руб/час) для зоны, **чтобы** цены на карте пересчитывались для новых/существующих мест. |
| A-03 | **Как** администратор, **я хочу** добавлять новое парковочное место на сетку, **чтобы** расширять вместимость (мок). |
| A-04 | **Как** администратор, **я хочу** удалять место (если оно свободно), **чтобы** исключить недоступные ячейки из продажи. |

## 4. Нефункциональные требования

- SPA на Vue 3 + Pinia + Vite; маршрутизация Vue Router; **ленивая загрузка** админ-раздела.
- Отзывчивая карта: мемоизация ячеек при частых обновлениях датчиков.
- Базовый **офлайн**: статика и приветствие через Service Worker (PWA).
- E2E: сценарий «запрос в чат → найти → забронировать → подтверждение».

## 5. Критерии приёмки (ключевые)

1. Статусы на карте соответствуют данным стора; смена статусов от mockApi видна в UI.
2. Запрос «найди у входа N» приводит к выделению места и сообщению в чате с кнопкой брони (если место свободно).
3. Бронь отображается в `bookingStore` и в UI «Моё бронирование».
4. Админ-роут грузится отдельным чанком (проверка в Network / тест lazy import).

## 6. Вопрос для отчёта (блок 1) — краткий ответ

AI-first чат добавляет на диаграммы потоков данных сущности **NLU/Intent Router**, **Orchestrator**, **Policy/Guardrails**, **LLM-Proxy** и цикл **язык → намерение → мутация состояния → UI** вместо только детерминированных экран→CRUD потоков. Подробнее см. `FINAL_REPORT.md`, раздел 7.
